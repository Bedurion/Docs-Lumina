Documentation · Website Publishing

Submit community media in Discord. Publish only after staff review.

Members use a private proposal thread, moderators review the submission and an administrator makes the final publication decision. The bot does not save uploaded image or video files on the Luminox computer.

Member flow

One private proposal for one media file

  1. Press Submit content in the Website panel.
  2. Enter a public title, description and alternative text. Credit is optional.
  3. Luminox opens a private thread for the author and eligible staff.
  4. Upload exactly one supported JPG, PNG or MP4 file, up to 10 MB.
  5. Press Ready for review when the description and media are final.

The proposal confirms that the guild has permission to publish the media and the selected credit. Links to external files and unsupported file types are rejected.

Staff review

Moderators review; administrators publish

Eligible moderators can approve, reject or request changes. Rejections and change requests require a reason so the author knows what to fix. Approval means “ready for the final check”; it does not publish the file immediately.

DraftThe author can still replace the file or edit the submission.
Pending reviewStaff check the content, credit and permission to publish it.
ApprovedAn administrator can make the final publication decision.
PublishedThe gallery entry is live and the proposal thread is archived.

Final publication

Publish from the approved thread

Use the Publish button or /website publish inside the matching approved proposal thread. Luminox sends the approved submission to the fixed publishing process; it does not accept a different repository, workflow or destination from a Discord form.

Before the file reaches the public gallery, the publishing process checks the real file type and size, removes hidden metadata, creates a safe web version and rejects unexpected output. Submitted text is displayed as text rather than interpreted as website code.

What the administrator needs to know

Approve only content the guild is allowed to publish. The technical file checks are automatic, but copyright, privacy and community suitability still require human review.

Safety

Limited input, private review and no credentials in Discord

  • One JPG, PNG or MP4 file per proposal, up to 10 MB.
  • Direct download links, archives, documents, scripts and executable files are rejected.
  • The original filename is never used as a public website path.
  • Uploaded file bytes are not stored in the Luminox data file.
  • Only the approved file can continue to publication.
  • Repository credentials are configured privately by the deployment owner.
  • Discord panels show only whether publishing is ready, never credential values.
  • Rejected, cancelled and completed proposal details are cleaned according to the server's retention policy.
Never paste credentials into Discord

Normal members, moderators and server administrators should not be asked for a bot token, repository token, environment file or private key during this workflow.

Administrator setup

Publish the panel after the private connection is ready

/website set channel:#website url:https://your-public-site.example/website overview/website refresh
  1. The deployment owner configures the repository connection privately, outside Discord.
  2. An administrator runs /website set with the panel channel and public HTTPS address.
  3. Run /website overview to see whether the panel and publisher report Ready.
  4. Use /website refresh after changing the public address or when the status needs a new check.

The overview reports missing setup by name but does not reveal stored values. Re-running /website set edits the existing panel instead of creating a duplicate.

Discord permissions

Panel channel and private proposal threads

  • View Channel, Send Messages and Embed Links.
  • Read Message History.
  • Create Private Threads and Send Messages in Threads.
  • Manage Threads so Luminox can add reviewers, lock and archive completed proposals.
  • Manage Messages if the panel channel is protected against normal chat.

The panel is edited in place when its status changes. Routine checks do not create another panel message or a new-message notification.
